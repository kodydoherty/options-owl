1. Problem — binary is the wrong shape

momentum_confirm is currently all-or-nothing: ON blocked so much it cost +45% in the backtest; OFF let in immediate-faders (2026-06-11 live: SMCI/MSTR/PLTR peaked +4–13% then stopped, −$368 combined). One gate is trying to do two jobs with one threshold — reject garbage AND don't reject runners. A single boolean can't separate a +4% fader from a +200% runner at entry.

Core idea: replace the binary momentum gate with a continuous entry-quality score that (a) hard-blocks only the clear immediate-fade signature and (b) feeds the rest into position sizing — "how much," not "yes/no." A weak-but-tradeable signal costs 1 contract, not a full-size −$158 stop; a runner-like signal gets real size.

2. Goals & success criteria

  • Reduce the immediate-fade loser rate (trades that peak <~15% then stop) vs momentum-off, WITHOUT cutting runner capture.
  • Beat BOTH binary-on and binary-off on risk-adjusted return (P&L + profit factor + max-DD) across multiple historical windows (OOS).
  • Convert the quality score into a sizing multiplier so conviction → size (unifies with the sizing overhaul).
  • Preserve / improve total P&L and runner count vs the current live config (momentum-off).

3. Design

3.1 Entry-quality score (0–1)

A continuous score computed at entry from serve-time-safe early features (already available or cheap to add): early follow-through (1–3 min premium/underlying momentum after the signal), price extension vs VWAP, candle structure (body/wick, direction agreement), RSI, early volume, spread, and the ML pattern-confidence. No future data.

3.2 Two outputs from one score

  1. Hard floor (reject): below a calibrated quality floor → block (the clear immediate-fade signature — today's +4% peakers). This is the narrow "reject garbage" job, NOT the blunt momentum_confirm.
  2. Sizing multiplier: above the floor, map quality → budget% (monotonic). Low-but-tradeable → small (e.g. 1 contract / 0.3× slot); runner-like → scale up (toward the position cap). Replaces the backwards 60% confidence tier and composes with the sizing experiment's curve.

3.3b Reversal-confirmed entry on LIVE TICKS (the patient-entry upgrade)

VALIDATION RESULT (2026-06-11): REFUTED on real tick data — do not build as-is. Tick analysis of ~2.5 wk / 92 executed trades (all 5 bots) found counter-trend is NOT the fader cause (against-trend trades won MORE: 66% vs 51%; half the faders faded WITH the micro-trend), and simulated reversal-confirmed entry either missed winners or failed to filter faders (waiting pays up via theta+continuation). Direction does not separate faders. Tiny sample (caveat), but combined with confidence-scaling also overfitting OOS, the working conclusion is: faders likely can't be filtered at entry. Re-scope this spec away from entry-filtering toward (a) runner-PREDICTION for sizing-up and (b) exits/sizing. Section kept for the record.

Instead of binary block (momentum_confirm) or blind wait-then-buy (current dip-confirm), a good signal that's momentarily counter-trend enters a patient-wait state and only fires when the underlying reverses into our direction — capturing a better price AND trend alignment. This is what turns "don't fight the trend" from a rejection into a timing edge.

  • Use live tick/quote stream, NOT 1-min candles. Current dip-confirm polls premium every ~5s and derives VWAP/support from candles — both lag up to a full bar. Detect the reversal on the real-time Polygon WS tick feed (already running via market_data_stream / FlowCollector): underlying up-ticks off a local low, micro higher-low, VWAP reclaim, short-EMA/RSI flip — sub-second, not a delayed candle close.
  • Reversal trigger (CALL): underlying stops falling and turns up (N consecutive up-ticks / higher-low / reclaim of a micro level) → enter near the local bottom with momentum now aligned. (PUT: mirror.)
  • Quality-scaled patience: a high-quality signal gets a longer wait window to find its reversal; a weak one gets little/none. Cap the window — if no reversal in the window, SKIP (this is how we avoid the fader instead of buying into it).
  • Makes both gates smart: anti_chase → "chasing is OK if a live-tick reversal confirms" (so we still catch runners on the turn); momentum_confirm → "wait for momentum to flip into our direction" rather than a hard block.
  • Risk: a fast reversal-then-rocket could trigger before confirmation (miss). Mitigate with a fast tick trigger + quality-scaled patience. Tune the confirmation strictness on data.
Backtest-fidelity caveat: ThetaData history is 1-min bars, so the backtest can validate the concept (reversal-confirmed entry on 1-min) but the live tick implementation is finer — expect live to differ from backtest on this component; validate the live version with shadow/paper before full size.

3.3 Calibration (empirical, not hand-tuned)

Learn the quality score from the runner-vs-fader feature analysis on the backfilled ~2.75 yr: for trades that became runners (peak ≥100% / ≥200%) vs immediate-faders (peak <15% → stop), find which entry features separate them, then encode that as the score (logistic/GBM on entry features → P(runner-or-good) ). The floor + sizing curve are set from this distribution, then validated OOS — not guessed.

5. Testing protocol

  1. Runner-vs-fader feature analysis (prerequisite) → identifies the predictive early features + the fade signature.
  2. Implement the quality score + graduated gate/sizing behind backtest flags.
  3. Backtest A/B/C: smart-graduated vs binary-on (current code default) vs binary-off (current live). Metrics: total P&L, profit factor, max-drawdown, immediate-fade loser rate (peak<15%→stop), runner count (≥100%/≥200%), avg/95th position size.
  4. Multi-window walk-forward (month-by-month, 2024 → 2026) + dedicated OOS window. Reject in-sample-only winners (same discipline that caught put_bearish / the regime-0.19 threshold).
  5. Confirm the smart version beats BOTH binaries on risk-adjusted return before any live change.

6. Risks & rollback

  • Real-money entry AND sizing change — highest-leverage, highest-risk surface. Deploy only after multi-window + OOS clears; deploy via rebuild.sh; keep .env/config rollback.
  • Overfit risk — the quality score is fit on historical runners/faders; weight OOS heavily, prefer a simple robust model over a fragile high-AUC one.
  • Train/serve skew — the entry-quality features MUST be computed identically in trainer + live serving (use the shared-feature-module pattern). A unit parity test is mandatory.
  • Interim: today's live config (momentum-off) stands until this is validated; the binary revert is available as a fallback if live losers persist.

7. Sequencing

Queued behind the in-flight work: multi-year backfill (running) → runner feature-analysis → THIS (entry-quality score) ⊕ conviction sizing → backtest + multi-window/OOS → deploy. This is the convergence of the three priorities: catch runners, size to conviction, stop bleeding on faders.
