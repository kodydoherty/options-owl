1. Hypothesis

Twitter/X and other social media platforms (StockTwits, Reddit r/wallstreetbets, FinTwit) contain real-time options trade callouts from profitable traders that can serve as an additional signal source alongside the existing Discord (Neverland Pirates) collector. By ingesting and scoring social media callouts through the same ML pipeline, we can increase signal volume, diversify signal sources, and potentially capture moves that Discord misses.

Expected outcome: Social media signals, when filtered through the existing ML scoring pipeline (score ≥ 78), will produce a win rate within 5% of Discord signals and increase total daily signal volume by 30–50%. Combined portfolio P&L should improve due to more opportunities, not lower quality.

2. Background & Motivation

Currently, OptionsOwl relies on a single signal source: the Neverland Pirates Discord server. This creates concentration risk — if the Discord goes offline, changes quality, or stops posting, the bot stops trading entirely. Social media platforms offer a massive volume of options callouts from thousands of traders, but the signal-to-noise ratio is extremely low. The key challenge is extracting actionable, parseable trade signals from noisy social media text and filtering them through our existing quality gates.

Why Twitter/X Specifically

  • Real-time: Tweets arrive faster than Discord messages in many cases
  • Volume: Thousands of options callouts per day from FinTwit accounts
  • Track records: Accounts like @unusual_whales, @OptionsAction, @0DTETrader publish live calls with verifiable histories
  • API access: Twitter API v2 supports filtered streams with keyword/account matching
  • Sentiment layer: Aggregate sentiment can validate or contradict individual signals

7. Risks & Considerations

  • Signal-to-noise ratio: Twitter is ~95% noise. Without aggressive filtering, the bot will be overwhelmed with garbage signals. The ML pipeline (score ≥ 78) is the primary defense, but parsing failures waste compute.
  • API cost: Twitter API v2 Basic tier is $100/mo. Pro tier ($5,000/mo) is overkill. Basic provides 10K tweets/mo read + filtered stream — should be sufficient with account-based filtering.
  • Pump-and-dump risk: Social media is rife with coordinated pump schemes. Small-cap/meme tickers called on Twitter may be traps. Mitigation: only trade our existing 14-ticker universe (SPY, QQQ, NVDA, TSLA, etc), ignore unknown tickers.
  • Latency: By the time a tweet is posted, parsed, scored, and executed, the move may already be priced in. Need to measure signal-to-execution latency and compare to Discord path.
  • Account credibility decay: Good accounts go bad. Need rolling accuracy tracking with auto-mute below 40% WR threshold.
  • Rate limits: Twitter API v2 Basic has strict rate limits. Filtered stream is the most efficient approach (push vs poll), but reconnects count against limits.
  • Legal/TOS: Automated trading based on social media signals is legal, but scraping without API access violates Twitter TOS. Must use official API only.
  • Concentration risk reduction: The main strategic value is reducing dependence on a single Discord server. Even if Twitter signals are slightly worse quality, diversification has value.

8. Tweet Parsing Examples

Common formats the parser must handle:

# Format 1: Standard callout
"$NVDA 130C 6/6 @ 2.50"
→ ticker=NVDA, direction=CALL, strike=130, expiry=2026-06-06, premium=2.50

# Format 2: Verbose
"Bought NVDA 130 calls June 6 for $2.50"
→ ticker=NVDA, direction=CALL, strike=130, expiry=2026-06-06, premium=2.50

# Format 3: Shorthand
"TSLA 180p 0dte"
→ ticker=TSLA, direction=PUT, strike=180, expiry=today, premium=None (lookup)

# Format 4: Unusual Whales style
"🐳 NVDA | Unusual Call Activity | $130 Strike | 6/6 Exp | 5K Volume"
→ ticker=NVDA, direction=CALL, strike=130, expiry=2026-06-06, premium=None

# Format 5: Emoji-heavy
"🚀 $TSLA calls 180 LFG 🔥🔥🔥"
→ ticker=TSLA, direction=CALL, strike=180, expiry=None (infer 0DTE), premium=None

# Format 6: PUT callout
"$SPY puts looking good, grabbed 540p 0dte at .85"
→ ticker=SPY, direction=PUT, strike=540, expiry=today, premium=0.85

# Reject: Commentary (not a trade entry)
"NVDA looking strong today, might break 130 resistance"
→ REJECT (no specific trade entry)

9. Architecture Decision: LLM vs Regex Parser

ApproachProsConsCost
Regex patterns Fast, free, deterministic, no API dependency Brittle, can't handle novel formats, high maintenance $0
Claude Haiku classifier Handles any format, extracts structured data reliably, classifies intent API latency (~200ms), cost per call, dependency on Anthropic API ~$5–15/day at 200 tweets/day
Hybrid: regex first, LLM fallback Fast for common patterns, robust for edge cases, cost-efficient More complex code, two code paths to maintain ~$2–5/day
Recommendation: Start with hybrid approach. Regex handles the 80% of tweets that follow standard "$TICKER strike direction" patterns. Claude Haiku handles ambiguous tweets that regex can't parse. Both output the same TradeSignal schema.
