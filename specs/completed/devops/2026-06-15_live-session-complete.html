First Live Session — Bugs & Fixes

2026-06-15 was the first real live-market day for the V7 + UW flow system. The flow book — the new high-conviction Unusual Whales sweep source deployed the day before — had never placed a single live trade before this session.

Executive summary. The flow system was deployed 2026-06-14, but on the first live day it placed zero flow trades: a chain of independent gates was silently rejecting every flow signal — the WS decoder dropped 100% of alerts, the stop-price gate hard-rejected stop-less flow signals, the ML-CALL morning cutoff killed all-day flow calls, and the score-floor neutralized the deployed 0.62 ML threshold. On 2026-06-15 each blocker was found and fixed in sequence, and the flow book placed its first-ever live trade, #266 (SPY put). That trade then surfaced two more bugs — flow was trading the whale's far-dated expiry instead of nearest-DTE, and strike resolution was hammering Polygon directly — both now fixed. Alongside the flow chain, several ML/data fixes shipped: a Polygon-expiry fallback for ML scans, a GEX DataError, and ML-scan false timeouts.

1.Score-floor neutralized the 0.62 gate Fixed

Symptom The deployed ML_PATTERN_THRESHOLD=0.62 was supposed to admit ML signals in the 0.62–0.75 band (≈1.8× volume), but those signals were never actually traded — they were silently rejected downstream during position sizing.
Root cause score_to_contracts in risk/vinny_strategy.py carried its own independent floors: _SCORE_FLOOR=75 (≈ pattern 0.75) and _MIN_ML_CONFIDENCE=0.70. Any signal admitted by the 0.62 pattern gate but scoring below these floors was dropped before a contract count was ever computed. The two thresholds were set in different places and fell out of sync when 0.62 was deployed.
Fix Aligned the sizing floors with the deployed gate: _SCORE_FLOOR=62 and _MIN_ML_CONFIDENCE=0.62 in score_to_contracts (risk/vinny_strategy.py). The PUT confidence floor stays at 0.65 (PUTs are structurally worse and keep the tighter floor).
Status Fixed — Deployed mid-session, but after the 90-minute CALL scan window (5–90 min after open) had already closed, so the payoff is realized at the next open, not this session.

2.ML scan Polygon fallback used the wrong expiry Fixed

Symptom Several names returned 0 contracts on the ML scan and were effectively blind: JPM, MSTR, NFLX, PLTR, AMD, SMCI, BA.
Root cause fetch_live_option_chain in sourcing/ml_pipeline.py only queried today and tomorrow for an expiry. Names that don't have a Monday 0DTE (weekly-only or MWF tickers off-schedule) had no contract in that 2-day window, so the chain fetch returned zero rows.
Fix Added a bounded near-expiry fallback in fetch_live_option_chain that walks forward ~5 business days until it finds a tradeable expiry — the same near-expiry pattern smart-entry already uses.
Status Fixed — the previously-blind weekly/off-schedule names now resolve a chain.

3.UW flow WebSocket dropped 100% of alerts FixedCritical

Symptom Every live UW flow alert was rejected at decode time — zero flow signals ever reached the pipeline. The backtest had run clean, so this never surfaced until the live WS was connected.
Root cause The live flow WebSocket message has no type, strike, or expiry fields — they're encoded inside the OCC option_chain symbol, e.g. MSTR260618C00115000. The backtest used the UW REST endpoint, whose payload does include a type field, so the discrepancy was invisible offline. evaluate_flow_alert did if opt_type not in ("put","call"): return None — with type absent, opt_type was empty and every alert was rejected.
Fix When type is absent, parse the OCC option_chain symbol via parse_occ_ticker to recover option type / strike / expiry (collectors/uw_flow_collector.py, evaluate_flow_alert).
Status Fixed — live WS alerts now decode into TradeSignals. This unblocked the first gate in the chain; alerts then hit fixes #4 and #5 downstream.

4.stop_price gate blocked 100% of flow FixedCritical

This was THE reason the flow book never placed a trade. The flow column in paper_trades was 0 on every prior day because of this gate.
Symptom Even flow alerts that decoded correctly and were affordable were rejected at entry-pipeline gate 3. Example: an $896k META call at 14.3% risk died on the stop-price gate.
Root cause Flow signals carried stop_price=None. StopPriceGate in risk/pipeline.py (and a second guard in paper_trader.py) hard-reject any signal without a stop. ML signals pass this gate only because bot_runner assigns them a stop — entry×0.995 (call) / entry×1.005 (put) — but flow signals never had a stop assigned.
Fix _on_flow_signal now assigns flow signals the same 0.5% underlying stop computed from the resolved spot price (call: spot×0.995, put: spot×1.005), so they clear StopPriceGate. The stop is nominal — flow positions still exit via the V7 FSM, not the underlying stop.
Status Fixed — flow signals now pass gate 3. Combined with #3 and #5, this is what let trade #266 reach Webull.

5.Morning cutoff blocked flow CALLs after 11:00 ET Fixed

Symptom Flow CALL signals after 11:00 ET were rejected. Live proof: a $6.25M SPY whale call at 12:42 ET was killed by the morning cutoff.
Root cause TimeOfDayGate has a morning cutoff that only admits ML CALLs placed before 11:00 ET — an ML-CALL-specific rule (backtested: only pre-11:00 ML calls are profitable). But flow is an all-day source (whale sweeps fire anytime), and the cutoff was being applied to flow calls too.
Fix Flow signals now bypass the morning cutoff and the early-morning score bump in TimeOfDayGate (consistent with the existing flow gate-bypass set). The hard EOD 3:55 PM cutoff still applies to flow — flow does not trade into the close.
Status Fixed — regression covered in tests/test_scalp_target_and_morning_cutoff.py (flow-bypass cases).

6.Flow traded the whale's far-dated expiry instead of nearest-DTE FixedValidated

Symptom Trade #266 (the first live flow trade) was a 45-DTE SPY put that barely moved — low gamma, almost no premium response to the underlying. The V7 exits are tuned for 0DTE / short-DTE convexity and had nothing to work with.
Root cause _resolve_flow_strike used fs.expiry — the whale's OCC expiry, which is frequently far-dated (whales buy time). The backtest always used nearest-DTE (dte0 = min DTE across the chain), so it never exercised the far-dated path. Multi-day expiries were never tested at all because the thetadata backtest DB only contained 0–4 DTE.
Fix Flow now ignores the whale's expiry entirely and walks today → next business days to the nearest tradeable expiry. If it can't resolve one, it skips the signal — it never falls back to the whale's far-dated contract.
Validation Downloaded and backtested the whale's actual multi-day contracts to prove the call:
  • scripts/download_multiday_flow.py — 495k bars across 646 multi-day contracts.
  • scripts/backtest_multiday_flow.py — multi-day flow is a decisive loser.

7.Selective OTM strike strategy ValidatedWatch

Summary An A/B backtest (scripts/backtest_flow_otm_test.py) compared ATM vs cheaper OTM strikes on the same whale alerts and the same V7 exits.
Selector A per-ticker + per-month filter (must beat ATM PF, ≥60% of months positive, n≥15) was applied. Result: deploy OTM only for:
  • Calls: AMD, INTC, META, SPY
  • Puts: TSLA
Wiring Gated behind ENABLE_FLOW_OTM_STRIKE (true on all bots), with FLOW_OTM_CALL_TICKERS / FLOW_OTM_PUT_TICKERS and FLOW_OTM_TARGET_PREMIUM=2.0. The selection is a pure, tested function: bot_runner.select_flow_strike.
CAVEAT (WATCH). The backtest prices off close with a 3% haircut and does not model OTM's wider bid-ask spread — so live OTM fills will be worse than the backtest implies. The v6_spread_gate (>15%) protects against the worst OTM spreads and will reduce OTM trade count. MU OTM was a net loser → stays ATM (MU remains in the whitelist, auto-gated by per-trade risk as the account grows).
Status Validated (backtest) Watch (real fills) — deployed for the 5 selected ticker/side combos.

9.Flow strike resolution moved to Redis-first DeployedWatch

Symptom Flow was hitting Polygon directly on every signal — a congestion source, and the direct cause of trade #266's chain-fetch timeout.
Root cause Flow strike resolution called polygon_option_chain + fetch_live_underlying_price on each signal, bypassing the harvester's already-cached data and adding load to a Polygon endpoint that was already congested during the live session.
Fix Strike resolution now reads spot + chain from the harvester's Redis: get_price for spot and get_option_snapshots_for_ticker for the chain (ATM ± 10%, via HARVEST_STRIKE_WINDOW=5). Polygon is only hit on a Redis miss. Webull 429s are a separate problem — per-account order-status polling, which can't be centralized.
Status Deployed Watch — validates at the next open when the harvester repopulates Redis with fresh snapshots.

10.GEX DataError Fixed

Symptom GEX regime features silently defaulted to 0 — 128 warnings before the fix, 0 after. The underlying gex_ticks table was healthy (55k rows), so this was not a data-availability problem.
Root cause regime_features._load_serving_gex passed a string as the query parameter for a timestamptz column (captured_at). asyncpg raised a DataError, which the function's generic except caught and mislabeled as "table unavailable" — so GEX features fell back to 0 even though the table was fine.
Fix Pass a datetime (not a string) for the captured_at timestamptz param in _load_serving_gex.
Status Fixed — GEX regime features now populate from the live table; warning count dropped 128 → 0.

11.ML scan false timeouts Fixed

Symptom ML scans for MSTR / PLTR / NFLX were killed by an outer timeout even when the underlying Polygon fetches were succeeding — successful work was being thrown away.
Root cause A Redis-miss ticker performs 2 sequential Polygon snapshot fetches at up to 10s each. Two successful 10s fetches sum past the _scan_ticker_safe outer timeout of 15s, so the wrapper aborted the whole scan even though both fetches would have completed.
Fix Raised the _scan_ticker_safe outer timeout 15s → 25s, giving room for two sequential 10s fetches plus overhead. The inner 10s per-fetch caps remain, so a genuinely hung call still can't stall the loop.
Status Fixed — Redis-miss tickers complete their scans instead of being falsely timed out.

Gold-standard parity

After this session, production matches the gold-standard V7 flow logic:

  • Nearest-DTE strike/expiry resolution (whale expiry ignored).
  • Whitelists aligned (incl. SPY in the put universe — fix #8).
  • Conviction sizing with p_runner=None (runner tilt still off pending live feature validation).
  • All-day flow (morning cutoff bypassed for flow — fix #5).
  • V7 exits on flow positions.

Two intentional differences from the backtest remain:

  1. OTM layer — a validated addition on top of the gold standard (fix #7), not present in the base backtest.
  2. Account-scaled sizing — prod sizes off the live account balance, vs the backtest's fixed-$750-per-trade edge measure.

scripts/flow_gold_standard_report.py was updated to mirror prod — it now uses select_flow_strike + the OTM layer and documents the account-scaled vs fixed-$ sizing difference, so the report and the live bots compute the same way.

↑ top

Data-flow architecture note

The intended architecture centralizes market data in the harvester: 

harvester  →  prices / premiums / option snapshots / candles  →  Redis  →  bots read

The gap exposed this session was that two paths bypassed the harvester and hit Polygon directly: the flow strike-resolution path and smart-entry. Flow is now fixed to be Redis-first (fix #9), reading spot via get_price and chain via get_option_snapshots_for_ticker, with Polygon only on a miss. Webull remains inherently per-account (order-status polling can't be centralized), so Webull 429s are tracked separately from the Polygon-congestion class of problem.
