1. Discovery: Streaming Was Already Working

Key finding (2026-06-01): The FlowCollector in the harvester ALREADY streams real-time option quotes via Polygon WebSocket and publishes full snapshots (bid/ask/greeks) to Redis every 1 second. The data was there all along — the ML scanner just never read it for PUTs.

The original diagnosis was that the scanner had a 60-90 second latency due to REST polling. Investigation revealed the FlowCollector WebSocket was providing 1-second freshness for ALL option types (calls AND puts) across 26 tickers. The actual bugs were:

  1. OWL-011 (FIXED): Scanner hardcoded "call" in Redis key lookup. PUT premium data was in Redis but never read. Fixed 2026-06-01.
  2. Remaining: Scanner tracks only CALL time series in TickerScanState. PUT-specific premium patterns cannot be detected by the ML model.

2. Current Architecture (Already Streaming)

WORKING NOW: Polygon Options WS -(real-time)-> FlowCollector -(1s flush)-> Redis snapshots Scanner -(15s poll)-> Redis GET owl:snapshot:{T}:{type}:{K}:{exp} -> ML -> Trade FlowCollector subscribed to 26 tickers (T.O:* trades + Q.O:* quotes) Publishes: owl:snapshot (full greeks), owl:option (premium only) Both CALL and PUT data available in Redis

2.1 What's Working

  1. FlowCollector WS connected to wss://socket.polygon.io/options — 26 tickers
  2. _process_quote() parses bid/ask, computes greeks (BS model), buffers full snapshot
  3. _redis_quote_flush_loop() publishes to Redis every 1 second
  4. Both CALL and PUT snapshots available via owl:snapshot:{ticker}:{type}:{strike}:{expiry}
  5. Scanner reads from Redis every 15 seconds — gets 1s-fresh data
  6. OWL-011 fix enables PUT premium lookup when direction=PUT (deployed 2026-06-01)

4. Polygon WebSocket API Details

4.1 Option Quote Channel

# Subscribe to option quotes
ws.subscribe("Q.O:O:AAPL260601C00310000")  # AAPL $310 Call, June 1 2026

# Message format:
{
  "ev": "Q",           // event type
  "sym": "O:AAPL...",  // option symbol
  "bp": 3.30,          // bid price
  "bs": 59,            // bid size
  "ap": 3.60,          // ask price
  "as": 47,            // ask size
  "t": 1780327394919   // timestamp (ms)
}

4.2 Option Trade Channel

# Subscribe to option trades
ws.subscribe("T.O:O:AAPL260601C00310000")

# Message format:
{
  "ev": "T",
  "sym": "O:AAPL...",
  "p": 3.45,           // trade price
  "s": 10,             // trade size
  "t": 1780327394919
}

4.3 Contract Symbol Format

# O:{TICKER}{YYMMDD}{C|P}{STRIKE*1000}
# Examples:
O:AAPL260601C00310000   # AAPL $310 Call, June 1 2026
O:AAPL260601P00310000   # AAPL $310 Put, June 1 2026
O:SPY260601C00756000    # SPY $756 Call, June 1 2026

4.4 Subscription Limits

Polygon Business plan: up to 1,000 concurrent WS subscriptions. With 16 tickers x 2 types (call/put) x 3 strikes (ATM +/-1) x 2 expiries = ~192 subscriptions. Well within limits.

7. Risks and Mitigations

  • WS disconnects: Auto-reconnect with exponential backoff. Fall back to REST polling. Log gap duration.
  • Data flood: Throttle ML evaluation to max once per 5s per ticker. Buffer quotes, process latest only.
  • Redis pub/sub reliability: Pub/sub is fire-and-forget. If a bot misses a message, the next one arrives in <1s. Snapshot keys remain as backup.
  • Polygon rate limits: WS has no rate limit on reads (it's a push channel). Only subscription changes count against limits.
  • ATM strike shifts: Monitor underlying price; when ATM shifts, unsubscribe old strikes, subscribe new ones. Do this at most once per minute to avoid thrashing.
