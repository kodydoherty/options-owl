1. Problem Statement

Impact: 100% of PUT signals from the ML scanner are rejected by smart entry due to premium deviation. On 2026-06-01, 8 PUT signals were detected but ALL rejected. Zero trades executed.

The ML scanner in bot_runner.py hardcodes "call" when fetching option snapshots from Redis. When the ML model detects a bearish pattern and emits a PUT signal, the premium attached to the signal is the CALL premium (e.g., $0.26 for a deep OTM call), not the PUT premium (e.g., $3.45 for an ITM put). Smart entry then fetches the actual PUT premium and sees a 100-1000% deviation, blocking the trade.

2. Root Cause

2.1 The Hardcoded CALL Key

# bot_runner.py line 211
contract_key = f"{ticker}:call:{strike}:{expiry}"  # ALWAYS "call"

2.2 ATM Resolution Also CALL-Only

# bot_runner.py line 228
calls = [s for s in snaps if "call" in s.get("contract_key", "").lower()]

2.3 Direction Determined AFTER Data Collection

The scanner accumulates CALL premium data into the state arrays. The ML model runs on this CALL data. Direction (CALL vs PUT) is determined after the model fires, by checking underlying price movement (lines 712-719). But the premium in the signal is still the CALL premium from state.asks[-1].

3. Fix Design

3.1 After Direction is Determined, Fetch Correct Option Type

Once the ML model fires and direction is determined (line 712-719), if direction is PUT:

  1. Fetch the PUT snapshot from Redis using {ticker}:put:{strike}:{expiry}
  2. If Redis has no PUT data, try Polygon REST fallback
  3. Use the PUT premium (ask) as the signal premium
  4. If PUT data unavailable, use the CALL premium as fallback (preserves existing behavior)

3.2 Files to Modify

FileChange
options_owl/bot_runner.py Add _fetch_put_snapshot_from_redis(); in _run_ml_for_ticker(), after direction is determined as PUT, fetch PUT premium and override signal premium

3.3 Why Not Track Both CALL and PUT States?

Tracking dual states doubles memory, Redis reads, and complexity. The pattern model was trained on generic option premium patterns (not direction-specific). The simpler fix is: detect pattern on CALL data (working as trained), then fetch the correct premium for the actual direction before emitting the signal.

4. Testing

  • Unit test: verify _run_ml_for_ticker returns PUT premium when direction=PUT
  • Integration test: PUT signal with Redis PUT data available passes smart entry
  • Edge case: PUT data not in Redis, falls back to CALL premium
  • Manual: deploy, wait for PUT signal, verify smart entry deviation is reasonable

5. Secondary Bug: Score Floor Mismatch

Also discovered: Pipeline approves at score ≥ 75, but score_to_contracts() rejects at score < 78. The one IWM CALL signal that passed all gates was killed by sizing: SIZING: score 75 < 78 → 0 contracts (rejected)

The pipeline score gate and the sizing score floor should match. Either raise the pipeline gate to 78 or lower the sizing floor to 75.
