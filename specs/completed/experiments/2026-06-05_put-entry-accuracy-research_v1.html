1. Problem Statement

On bearish days (like 2026-06-05), the ML scanner finds valid PUT signals all day but ENABLE_PUT_TRADING=false blocks every one of them. Meanwhile, no CALL signals pass the pattern threshold because the market is going down. Result: zero trades on a perfectly tradeable day.

The core question: Can we profitably trade PUTs using the same ML + candle logic that works for CALLs, or does the ML actually hurt PUT entry and we should rely purely on candle/logic gates?

Revenue impact: Every bearish day with zero trades is missed profit. With 16 tickers being scanned, a typical bearish day produces 5-15 PUT signals scoring 74-80. If even 30% of those are profitable at the PUT scalp config (+50% target, -60% stop), that's significant alpha being left on the table.

2. Current Architecture Analysis

2.2 Critical Flaws Identified

#FlawImpactSeverity
1 Pattern model trained on CALL data only. It was trained to predict "will this CALL option gain 20%+ in 120 minutes?" The features (premium momentum, IV expansion, delta movement) have opposite meanings for PUTs. A PUT gaining value means underlying is dropping — the exact scenario where CALL premiums decline. Model confidence for PUTs is essentially random — it's measuring CALL favorability and then the direction is flipped based on a simple 0.15% threshold. HIGH
2 Direction decision is too simple. Underlying down 0.15% from open is the only trigger. No momentum confirmation, no trend strength, no multi-timeframe alignment. A stock that gapped down 0.3% at open but is recovering will still trigger PUT. False PUT signals on mean-reversion days. HIGH
3 ML models see CALL chain data but signal is labeled PUT. The entry_timing model has an is_call feature but still receives CALL option chain metrics (CALL premium momentum, CALL delta, CALL IV). The feature vector is incoherent. Entry timing confidence is meaningless for PUT signals. MEDIUM
4 Backtest uses PUT chain data correctly, but production doesn't. In backtest_gold_standard.py:1859-1868, the pattern model is run on put_ticker_data (actual PUT option chain). But in production, the scanner only tracks CALL chains in TickerScanState and swaps in PUT premium at the end. Backtest results for PUTs don't match production behavior. HIGH
5 No PUT-specific pattern model exists. We have V2 direction models (14 per-ticker PUT models) but these are secondary validators, not primary sourcing models. The main pattern_entry model is direction-agnostic by design but trained on directional data. No model is specifically optimized for "will this PUT option gain value?" MEDIUM

2.3 What the Backtest Does Differently (and Correctly)

The gold standard backtest (scripts/backtest_gold_standard.py) handles PUTs better than production:

  • Loads separate put_ticker_data from harvester — actual PUT chain snapshots (premium, IV, delta, greeks)
  • Runs the pattern model on PUT chain features directly (lines 1859-1868)
  • Uses check_spy_put_gate() to verify SPY is actually bearish before allowing PUTs
  • Has separate PUT concurrent slot limits (2 normal, 4 bear mode)
  • Uses PUT_SCALP_CONFIG for exit management
Key insight: The pattern model's features (premium momentum, bid/ask dynamics, IV, volume) are actually direction-agnostic — they measure "is this option about to gain value?" When fed PUT chain data (where rising premium = underlying dropping), the model CAN work for PUTs. The problem is that production feeds it CALL data and then relabels the signal as PUT.

3. Proposed Approaches (Ranked by Effort & Impact)

Approach A: Fix Production to Match Backtest (Feed PUT Chain Data to ML)

Effort: Medium | Impact: High | Risk: Low

Make production scan PUT option chains alongside CALL chains, so the ML models see the correct data for the direction they're scoring.

Changes Required:

FileChange
options_owl/sourcing/ml_pipeline.py Add TickerScanState for PUT chains (separate state per direction). Populate from Redis PUT snapshots. Run pattern model on PUT features when underlying direction is bearish.
options_owl/bot_runner.py Dual-track scanning: maintain both CALL and PUT TickerScanState. When underlying is down, score both CALL and PUT, emit the stronger signal. Remove the post-hoc premium swap (lines 734-750).
options_owl/sourcing/data/harvester_options.py Already supports PUT queries. No changes needed.
options_owl/db/redis_client.py Already publishes PUT snapshots. Verify key format supports lookup by option_type.

How it works:

# Current (broken):
state = TickerScanState(...)  # CALL chain only
pattern_conf = model.predict(CALL_features)  # model sees CALL data
if underlying_down: direction = "PUT"  # relabel after the fact
premium = redis.get_put_premium()  # swap premium at the end

# Proposed (matches backtest):
call_state = TickerScanState(...)  # CALL chain
put_state = TickerScanState(...)   # PUT chain (separate)

# Score both when market direction is unclear
call_conf = model.predict(call_state.features)
put_conf = model.predict(put_state.features)

# Emit the direction with stronger confidence
if put_conf > call_conf and bearish_confirmation():
    emit_put_signal(put_state)
elif call_conf >= threshold:
    emit_call_signal(call_state)

Approach B: Logic-Only PUT Entry (Skip ML, Use Candle/Technical Gates)

Effort: Low | Impact: Medium | Risk: Medium

Bypass the ML models entirely for PUTs. Use a rule-based system with candle data, technical indicators, and market structure to determine PUT entries. This is simpler and avoids the "wrong model on wrong data" problem entirely.

Logic-based PUT entry gates (all must pass):

  1. SPY Direction Gate: SPY down 0.3%+ from open (confirms broad market bearish)
  2. Ticker Trend Gate: Underlying price below VWAP AND below 9 EMA on 5m chart
  3. Momentum Gate: Last 3 of 5 candles are red (close < open)
  4. RSI Gate: 5m RSI < 45 (bearish momentum) but > 25 (not oversold bounce territory)
  5. Volume Gate: Current volume > 80% of 20-period average (not thin/illiquid)
  6. Premium Gate: PUT premium between $0.30 and $6.00, bid-ask spread < 35%
  7. Time Gate: Between minute 5 and minute 60 (avoid open chaos + theta decay)
  8. No Double-Dip: Haven't already entered this ticker as PUT today

Advantages:

  • No model mismatch — logic gates are explicitly designed for bearish setups
  • Easy to backtest, tune, and explain
  • Can be deployed immediately (PutBearishConfirmGate already has 3 of these gates)
  • Works on days where ML models have no edge (novel market conditions)

Disadvantages:

  • No learning from historical patterns — static rules don't improve over time
  • Lower signal quality vs a properly-trained PUT model
  • More false signals from choppy/consolidation phases

Approach C: Train a Dedicated PUT Pattern Model

Effort: High | Impact: Highest | Risk: Medium

Train a separate LightGBM model specifically on PUT option chain data, predicting "will this PUT gain 20%+ in 60 minutes?" (shorter window than CALLs since PUTs are scalps).

Training data:

  • Source: options_data.db PUT chain snapshots from harvester (available since ~April 2026)
  • Label: PUT premium increased 20%+ within 60 minutes of snapshot
  • Features: Same as pattern_entry but computed from PUT chain (PUT premium momentum, PUT IV, PUT delta, PUT volume, underlying decline rate, VWAP distance)
  • Additional PUT-specific features:
    • underlying_decline_rate_5m — how fast underlying is dropping
    • put_iv_vs_call_iv — put/call IV skew (fear indicator)
    • put_volume_vs_call_volume — put/call volume ratio
    • vwap_distance_pct — how far below VWAP
    • spy_correlation_5m — is ticker moving with SPY?

Advantages:

  • Model actually learns PUT-specific patterns from real data
  • Can capture non-obvious relationships (IV skew preceding big drops, etc.)
  • Highest potential accuracy once trained

Disadvantages:

  • Need sufficient PUT training data (check harvester DB for PUT snapshot count)
  • PUT market dynamics differ from CALLs (faster moves, higher IV, wider spreads)
  • Risk of overfitting to recent bearish periods

Approach D: Hybrid (Logic Gates + ML Confirmation)

Effort: Medium | Impact: High | Risk: Low — RECOMMENDED FIRST STEP

Combine Approaches A and B: use logic gates as the primary PUT entry filter, then use the pattern model on actual PUT chain data as a confirmation/quality score (not a hard gate).

How it works:

# Phase 1: Logic gates (hard requirements)
if not spy_bearish(threshold=-0.3%):        return SKIP
if not price_below_vwap(ticker):            return SKIP
if not bearish_candle_pattern(ticker):      return SKIP
if not rsi_in_range(ticker, 25, 45):        return SKIP
if not volume_adequate(ticker):             return SKIP
if not premium_in_range(put_premium):       return SKIP

# Phase 2: ML quality score (soft — ranks signals, doesn't block)
put_state = get_put_chain_state(ticker)  # actual PUT data
ml_score = pattern_model.predict(put_state.features)
# Use ml_score for:
#   - Ranking multiple PUT signals (take highest scoring)
#   - Position sizing (higher score = slightly more contracts)
#   - Signal logging and post-hoc analysis

# Phase 3: Emit signal
emit_put_signal(ticker, premium, ml_quality=ml_score)

Why this is the best starting point:

  • Logic gates prevent bad entries even if ML is wrong
  • ML adds value as a ranker without being a single point of failure
  • Can measure ML contribution over time (does higher ML score = better P&L?)
  • If ML proves valuable, upgrade to Approach C (dedicated PUT model)
  • If ML adds nothing, simplify to Approach B (pure logic)

10. Conclusion & Decision

Verdict: ADOPTED — Approach A+C (Dual-Chain + Dedicated PUT Model)

Dedicated PUT model (AUC 0.8038, 27 features) + dual-chain scanning + CALL-style trailing (no profit target, no hold limit) is the winning configuration. PUTs add $+31K-63K on top of CALL profits with no degradation. Breakeven ratchet + trail system handles exits.

Key Findings

  • Dedicated PUT model beats CALL model on PUT data — 27 features including cross-chain (iv_skew, put_call_volume_ratio) outperform 15 generic features
  • Near-low labeling fixed the coin-flip problem — V1 had 54% positive rate (noise), V2 has ~10% (learnable signal)
  • Walk-forward validation confirms no overfitting — 5 folds, AUC 0.8026 ± 0.007
  • All-day scanning captures afternoon selloffs — CALL scan 5-90min, PUT scan 5-360min
  • No profit target is optimal — tight targets (15-20%) destroy value. Trail system handles exits perfectly
  • PUTs are additive — zero degradation of CALL P&L. Different market conditions captured
