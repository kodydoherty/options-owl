1. Overview

Eliminate ALL polling-based data paths during market hours. Every agent should receive stock prices and option premiums via WebSocket push through Redis, with sub-second latency. The current architecture has a mix of WS, REST polling, and yfinance fallbacks that creates unpredictable 2-120 second delays.

2. Problem Statement

Critical Issue: All agents see data up to 2 minutes delayed. This means exit decisions (stop losses, trailing stops) fire late, costing real money on every trade. A 2-minute delay on a 0DTE option losing $0.50/minute = $100 per contract in avoidable loss.

2.2 Hotfix Applied (2026-06-10)

  • Position monitor now checks Redis snapshots FIRST (sub-second) before falling back to REST
  • Harvester poll interval reduced from 60s to 15s (for open_interest/new strike discovery only)
  • Verified: Redis snapshots are 2s old during market hours (flow WS is connected)

2.3 Remaining Issues

  • REST poll still runs every 15s — should only run for discovery (new strikes, open_interest), not for premium data
  • No WS health watchdog — if flow WS drops mid-day, no alarm, no auto-recovery notification to agents
  • Boot race condition — if bot starts before WS connects, it gets stale/empty Redis data. Should retry until WS data flows
  • No end-to-end latency monitoring — we can't see "how old is the data the bot is actually trading on"
  • yfinance fallback still exists — slow, rate-limited, adds 3-5s when it fires. Should be removed for live bots

3. Target Architecture

Polygon /stocks WS ──→ candle_collector ──→ Redis prices     ──→ All bots (instant)
Polygon /options WS ─→ flow_collector ───→ Redis snapshots   ──→ All bots (instant)
                                          Redis premiums     ──→ Position monitor (instant)

Polygon REST (15s) ──→ harvester ────────→ SQLite (bulk)     ──→ Backtest/analytics only
                                          Redis (discovery)  ──→ New strikes, open_interest

3.1 Principles

  • WS is the primary data source. REST polling is ONLY for discovery (new strikes, OI). Never for premium/price data during market hours.
  • Redis is the distribution bus. Harvester writes, all bots read. One source of truth.
  • Every WS connection has a health monitor. If no messages for 30s during market hours, alert + force reconnect.
  • Boot-time resilience. Bots retry Redis reads every 5s until data is fresh. Don't crash, don't trade on stale data.
  • Latency is observable. Every premium read logs its source and age. Dashboard shows real-time data freshness.

5. WS Health Watchdog Design

# In harvester.py, as a background task alongside the poll loop:
async def _ws_health_watchdog(candle_collector, flow_collector):
    """Check WS health every 30s during market hours."""
    while True:
        await asyncio.sleep(30)
        if not _is_market_hours():
            continue

        issues = []
        if not candle_collector.ws_connected:
            issues.append("candle_ws DISCONNECTED")
        elif candle_collector.last_msg_age > 60:
            issues.append(f"candle_ws stale ({candle_collector.last_msg_age:.0f}s)")

        if flow_collector and not flow_collector.ws_connected:
            issues.append("flow_ws DISCONNECTED")
        elif flow_collector and flow_collector.last_msg_age > 60:
            issues.append(f"flow_ws stale ({flow_collector.last_msg_age:.0f}s)")

        if issues:
            logger.critical(f"WS HEALTH ALERT: {', '.join(issues)}")
            # Publish to Redis so all bots know
            await redis_client.set("owl:ws_health", json.dumps({
                "healthy": False, "issues": issues, "t": time.time()
            }), ex=120)
            # TODO: Send Discord DM alert
        else:
            await redis_client.set("owl:ws_health", json.dumps({
                "healthy": True, "t": time.time()
            }), ex=120)

6. Boot-Time Data Wait Design

# In bot_runner.py, before entering scan/monitor loop:
async def _wait_for_fresh_data(max_wait=300, check_interval=5):
    """Wait until Redis has fresh option snapshots before trading."""
    start = time.time()
    while time.time() - start < max_wait:
        try:
            snap = await redis_client.get_option_snapshot("SPY:call:*")
            if snap and snap.get("t") and (time.time() - snap["t"]) < 30:
                logger.info(f"Fresh data available (SPY snapshot age={time.time()-snap['t']:.0f}s)")
                return True
        except Exception:
            pass
        elapsed = time.time() - start
        logger.info(f"Waiting for fresh Redis data... ({elapsed:.0f}s / {max_wait}s)")
        await asyncio.sleep(check_interval)

    logger.warning(f"No fresh data after {max_wait}s — proceeding with caution")
    return False

8. Security Considerations

  • Redis data is untrusted — validate all numeric values before using for trade decisions
  • WS reconnect with backoff prevents thundering herd and Polygon rate limit bans
  • Health watchdog runs in harvester only (single writer), not in bots (avoids N×M alert storms)

9. Rollback Plan

Each component has independent fallbacks:

  • If Redis is empty/stale, position monitor falls back to Polygon REST (existing behavior)
  • If flow WS is down, harvester REST poll (now 15s) still publishes to Redis
  • Boot-time wait has a 5-minute timeout — bots always start eventually
