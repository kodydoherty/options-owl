1. Overview

The 2026-06-10 ML review proved that the regime model's real edge collapses to ~0.53 (coin-flip) once data leakage is removed, but recovers to a genuine walk-forward AUC of 0.636 when fed serve-time-safe cross-market and dealer-positioning (GEX) features. ticker_selection recovered from broken/NaN to 0.711. Those models cannot be deployed yet because the live serving path (ml_pipeline.py:compute_regime_features) only produces the old 18 features, and the GEX features were sourced from a stale external Unusual Whales sqlite (uw_historical.db, ends 2026-05-21) that is never fed live. This spec expands the harvester to self-compute and persist the missing feature inputs (GEX / charm / vanna) and introduces a single shared feature module used by both the trainer and the owlet serving path, so training and serving features are identical by construction.

2. Problem Statement

  • Train/serve skew (root cause): training computes features from historical sqlite DBs (thetadata + UW) while serving computes a different, smaller subset from Postgres/in-memory. They drift apart — this is the "40 vs 18" gap and the same class of bug as the V2 feature skew already fixed.
  • No live GEX source: dealer gamma/charm/vanna currently come from a stale external UW sqlite. At serve time they zero-fill, degrading the regime model.
  • Hardcoded data-provider credentials: scripts/download_thetadata.py falls back to hardcoded ThetaData creds instead of reading .env (pre-existing; flagged for cleanup here).

3. Goals & Success Criteria

  • Harvester computes charm + vanna (in addition to existing delta/gamma/theta/vega) and persists a per-ticker dealer-positioning aggregate (net GEX, charm, vanna, call/put gamma) to a new Postgres table every harvest cycle — no external UW dependency.
  • A single shared feature module produces the regime feature set; both scripts/train_ml_models_v3.py and ml_pipeline.py:compute_regime_features import it. A unit test asserts training features == serving features (zero skew).
  • The deployed regime + ticker_selection models receive their full feature set live (no silent zero-fill).
  • signal_quality reworked the same way regime/ticker_selection were (label + serve-time features + robust validation); no longer a leakage artifact.
  • Full local test suite green; models retrained on the 190-day dataset; gold-standard backtest re-run before any production deploy.

4. Business Logic

Dealer Gamma Exposure (GEX) is approximated from the option chain the harvester already snapshots, removing the need for a third-party feed.

4.1 Rules & Constraints

  • GEX proxy: per strike, gamma_exposure = gamma × open_interest × 100 × spot; net GEX = Σ(call_gamma_exposure) − Σ(put_gamma_exposure) over the snapshotted chain. Charm/vanna aggregated analogously from the new per-contract greeks.
  • Serve-time safety (CRITICAL): every feature must be computable from only past data at the model's serve time (~9:45 ET). No same-day full-day aggregates, no future bars. Open-interest is fixed at the open (prior-day close OI), so OI-based GEX is serve-time-safe.
  • Leakage features remain banned: day_range_pct, day_volume, and same-day options-volume aggregates stay dropped (asserted in code).
  • Single feature definition: a feature is defined exactly once in the shared module; trainer and serving call the same function against the same Postgres tables.
  • Underlying universe: SPY/QQQ/VIX candles are already harvested — cross-market features read from stock_candles; no new capture needed for those.

4.2 Edge Cases

  • Empty / sparse chain (illiquid ticker, pre-open) → GEX returns 0 with a logged reason, never NaN.
  • Missing OI (Polygon returns null) → contract excluded from the GEX sum, not treated as 0-gamma noise.
  • New gex_ticks table empty during rollout → serving feature module returns 0 for GEX features and logs once (model still loads).
  • VIX absent in some windows → feature 0-filled with a logged warning (handled by the missing-feature warner already added).

5. Technical Design

5.1 Architecture

Capture → persist → shared compute → serve. The harvester writes raw inputs (chain greeks incl. charm/vanna, OI, and a per-ticker GEX aggregate) to Postgres. A shared feature module reads those Postgres tables and computes the feature vector. Both the trainer (offline, may also read the historical thetadata sqlite via the same logic) and the owlets (online, read Postgres) call this one module.

6. Security Considerations

  • Remove hardcoded ThetaData creds from download_thetadata.py; read from env. Do not log secret values.
  • All new Postgres queries use parameterized placeholders (no string interpolation of ticker/date).
  • gex_ticks writes are server-side (harvester) only; no user input path.
Security checklist: input validation, parameterized SQL, secrets out of source, no secret leakage in logs.

9. Risks & Rollback

  • Live-bot risk: serving + harvester changes require a production deploy. Mitigate: full local test + gold-standard rerun first; deploy off-hours; journal/ (models) excluded from rsync so model swap is deliberate.
  • GEX proxy ≠ true dealer positioning (no dealer-sign data) — it is a directional approximation; validated by whether it lifts regime AUC out-of-sample, not by matching UW exactly.
  • Harvester write load: one extra small INSERT per ticker per cycle — negligible vs existing option_ticks batch.

Rollback Plan

Models are backed up at journal/models/ml_v3_backup_pre_walkforward/; the prior bug-fix deploy is commit f9497c2. Revert by restoring the old serving code + redeploying; the gex_ticks table and harvester writes are additive and harmless if unused.
