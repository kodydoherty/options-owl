1. Hypothesis

The current ML scan pipeline resolves the ATM strike once per day and never updates it, even when the underlying moves significantly. This causes the bot to trade stale, increasingly OTM options that bleed to zero via theta. By (A) dynamically re-resolving the ATM strike when the underlying drifts, and (B) adding a per-ticker OTM distance gate calibrated to each ticker's strike grid, we can eliminate bad OTM entries while preserving all valid ATM/near-ATM trades.

Expected outcome: Eliminate -60% to -90% OTM losers (like AMZN #249 on 2026-06-09: -$414) while preserving 95%+ of profitable trades. Net improvement: +$2K-5K per 100 trades by cutting deep OTM entries.

2. Background & Motivation

2.1 The Incident (AMZN #249, 2026-06-09)

  • AMZN strike $250 resolved when underlying was near $250 at market open
  • By minute 18 (9:48 ET), AMZN had dropped to $248.92 — strike was 0.43% OTM
  • DCA doubled down at $1.46 when premium was already fading
  • AMZN kept falling to $245 — $250 CALL went from $1.73 to $0.63 (-61%)
  • Grace backstop fired at -61%, total loss: -$414

2.2 Root Cause

  • _resolve_atm_from_redis() and find_atm_strike() in bot_runner.py pick the nearest strike to underlying — but only on the first scan
  • The strike is locked in TickerScanState.strike for the entire day (if call_state.strike <= 0: guard)
  • No pipeline gate checks if the strike is OTM at entry time
  • The PremiumGate only rejects penny premiums (<$0.20), not OTM distance

3.2 Test (Experiment) — Three-Layer Defense

The solution has three complementary layers that work together:

Layer 1: Dynamic ATM Re-Resolution (production only)

Track the underlying price when the strike was resolved. When underlying drifts >1% from that price, re-resolve the ATM strike from Redis and reset the scan state for that ticker. This ensures the bot is always tracking an option near the current underlying price. 

# bot_runner.py — new field in TickerScanState
strike_resolved_price: float = 0.0  # underlying when strike was resolved

# Re-resolve logic (every scan cycle, ~15s)
if call_state.strike_resolved_price > 0 and underlying_price > 0:
    drift_pct = abs(underlying - resolved) / resolved * 100
    if drift_pct >= 1.0:
        logger.info(f"ML_SCAN: {ticker} CALL drifted {drift_pct:.1f}%, re-resolving")
        call_state.strike = 0.0  # triggers re-resolution
        call_state.closes.clear()  # reset scan data for new strike
        ...

Layer 2: Per-Ticker OTM Gate (production + backtest)

Instead of a static % threshold, express the max OTM distance as a number of strike intervals. Each ticker gets a threshold calibrated to its strike grid. The gate computes: is the current underlying more than N strikes away from the trade strike? 

# Per-ticker strike intervals (from ThetaData research)
STRIKE_INTERVALS = {
    "SPY": 1.0, "QQQ": 1.0, "IWM": 0.5, "NVDA": 0.5, "MSTR": 0.5,
    "META": 2.5, "MSFT": 2.5, "TSLA": 2.5, "PLTR": 1.0, "AVGO": 2.5,
    "GOOGL": 2.5, "AAPL": 2.5, "AMD": 2.5, "AMZN": 2.5,
}

# Max strikes OTM per category
MAX_OTM_STRIKES = {
    "fine_grid":  3,   # SPY/QQQ/IWM/NVDA/MSTR: 3 strikes = 0.4-1.0%
    "standard":   2,   # META/MSFT/TSLA/PLTR/AVGO: 2 strikes = 0.8-1.5%
    "wide_grid":  1,   # AAPL/AMD/AMZN/GOOGL: 1 strike = 0.76-1.07%
}

# Gate logic
def otm_check(ticker, strike, underlying, direction):
    interval = STRIKE_INTERVALS.get(ticker, 2.5)
    category = get_strike_category(ticker)
    max_strikes = MAX_OTM_STRIKES[category]
    max_distance = interval * max_strikes

    if direction == "CALL":
        distance = strike - underlying
    else:  # PUT
        distance = underlying - strike

    return distance <= max_distance  # True = pass

4.3 Per-Ticker Threshold Configuration

# options_owl/risk/exit_v5/config.py (proposed)

STRIKE_INTERVALS: dict[str, float] = {
    # Fine grid (0.14-0.33% per strike)
    "SPY": 1.0, "QQQ": 1.0, "IWM": 0.5, "NVDA": 0.5, "MSTR": 0.5,
    # Standard grid (0.39-0.69% per strike)
    "META": 2.5, "MSFT": 2.5, "TSLA": 2.5, "PLTR": 1.0, "AVGO": 2.5,
    # Wide grid (0.76-1.07% per strike)
    "GOOGL": 2.5, "AAPL": 2.5, "AMD": 2.5, "AMZN": 2.5,
}

# Default for unknown tickers
DEFAULT_STRIKE_INTERVAL = 2.5

def get_max_otm_distance(ticker: str) -> float:
    """Return the maximum allowed OTM distance in dollars for a ticker.

    Fine-grid tickers (SPY, QQQ, IWM, NVDA, MSTR): 3 strikes OTM
    Standard tickers (META, TSLA, PLTR, etc.): 2 strikes OTM
    Wide-grid tickers (AAPL, AMD, AMZN, GOOGL): 1 strike OTM

    Returns: max dollar distance from underlying to strike.
    """
    interval = STRIKE_INTERVALS.get(ticker, DEFAULT_STRIKE_INTERVAL)

    FINE_GRID = {"SPY", "QQQ", "IWM", "NVDA", "MSTR"}
    WIDE_GRID = {"AAPL", "AMD", "AMZN", "GOOGL"}

    if ticker in FINE_GRID:
        return interval * 3   # SPY: $3, QQQ: $3, IWM: $1.5, NVDA: $1.5
    elif ticker in WIDE_GRID:
        return interval * 1   # AMZN: $2.5, AAPL: $2.5
    else:
        return interval * 2   # META: $5, TSLA: $5, PLTR: $2

5.2 Analysis

Static % thresholds are too aggressive because the backtest uses a fixed strike per day while production re-resolves dynamically. The backtest's fixed strike drifts from ATM as the day progresses, causing the OTM gate to block trades that in production would have been on a freshly-resolved near-ATM strike.

The per-ticker approach (V2) will fix this by expressing thresholds in strike intervals, which naturally accounts for each ticker's grid — and additionally the backtest should be upgraded to re-resolve strikes mid-day for true parity.

5.3 Unexpected Findings

  • Strike spacing varies 7.6x across tickers — any static % is fundamentally flawed
  • Trades within 0.1% of ATM generate 77% of all profit (from production paper_trades analysis)
  • Beyond 0.5% OTM, total P&L is negative — but 0.5% is less than one AMZN strike
  • The 1.5% threshold preserves 88% of trades with only 22% P&L reduction — suggesting most of the blocked trades were marginal, not catastrophic

6. Conclusion & Decision

Verdict: Modified — V1 deployed for production safety, V2 needed for optimal tuning

The dynamic ATM re-resolution (Layer 1) is the primary fix and is deployed now — it prevents the root cause of tracking stale strikes. The static OTM gate (Layer 2, v1) provides a safety net at 1.0% but needs to be upgraded to per-ticker strike-interval logic for optimal performance. The backtest needs multi-strike re-resolution (Layer 3) for accurate comparison.

7. Risks & Lessons Learned

  • Risk: Dynamic re-resolution resets scan state (closes, volumes, etc.), requiring the ML model to re-accumulate 5+ minutes of data on the new strike before it can fire. This creates a brief blind spot after each re-resolve.
  • Risk: Strike intervals may change over time (exchanges update grid spacing). The config should be verified quarterly against live chain data.
  • Lesson: Never use a single static number for something that varies 7.6x across tickers. Always check the actual data distribution before picking thresholds.
  • Lesson: Backtest limitations (fixed strike per day) can make valid production features look harmful. Always consider whether the backtest accurately represents production behavior before rejecting a feature based on backtest results alone.
  • Lesson: The DCA feature amplified the OTM loss by doubling down on a fading position. Consider adding OTM distance checks to the DCA logic as well — don't DCA into a position where the strike has drifted significantly OTM.
