1. Problem

Production ML scanner feeds CALL chain data to the pattern model, then relabels the signal as PUT based on underlying direction. The backtest correctly uses PUT chain data. This means production PUT signals have incoherent ML features (CALL premium/IV/delta/greeks when the signal is bearish).

2. Solution

Maintain a separate PUT TickerScanState per ticker, populated from Redis PUT snapshots. When underlying direction is bearish, run the pattern model on PUT chain features instead of CALL features. This matches what the backtest does.

4. Architecture (Before vs After)

Before (broken):

per ticker:
  1 TickerScanState (CALL chain data)
  → pattern_model(CALL_features) → if underlying down → label "PUT"
  → swap PUT premium from Redis at the end

After (correct):

per ticker:
  call_state = TickerScanState(CALL chain data from Redis)
  put_state  = TickerScanState(PUT chain data from Redis)

  # Determine direction from underlying price action
  direction = "PUT" if underlying down 0.3%+ else "CALL"

  # Run ML on the CORRECT chain data
  if direction == "CALL":
      signal = _run_ml_for_ticker(ticker, call_state, direction="CALL")
  else:
      signal = _run_ml_for_ticker(ticker, put_state, direction="PUT")

5. Key Design Decisions

  • Both states accumulate every minute — we don't know direction in advance, so we build both CALL and PUT timeseries. Only one is used for ML scoring per scan.
  • PUT state uses same ATM logic — find PUT strike closest to underlying price (ATM PUT, not OTM).
  • Direction threshold raised to -0.3% (from -0.15%) — stronger bearish confirmation, fewer false PUTs on choppy days.
  • entry_emitted is per-direction — can emit one CALL and one PUT per ticker per day (different states).
  • PUT still goes through PutBearishConfirmGate — the candle/VWAP/RSI gate is an additional safety layer on top of correct ML data.

6. Safety & Rollback

  • All existing PUT safety gates remain: PutBearishConfirmGate, PUT_BUDGET_MULTIPLIER=0.50, no DCA, PUT_SCALP_CONFIG, SPY bear gate, excluded tickers, 2/4 slot limits.
  • Rollback: Set ENABLE_PUT_TRADING=false — zero code changes needed.
  • Feature flag: ENABLE_PUT_DUAL_CHAIN=true — if false, falls back to old behavior (CALL data + premium swap).
